# Unified Recent-Change Indicator Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the "recent change" dot mean exactly one thing everywhere — a 14-day window over `trial_change_events` plus published intelligence notes, computed server-side, rendered by one neutral slate dot.

**Architecture:** A single SQL function `recent_change_window()` defines the window. `get_dashboard_data()` (trials) and `get_bullseye_assets()` (assets, rolled up from their trials) count change events + published intel within that window. The shared `ChangeBadgeComponent` loses its red/slate priority split and becomes a neutral dot; its text logic is extracted into a pure, unit-tested module. The bullseye tooltip and `BullseyeAsset` model are rewired off the old marker-based activity fields onto the unified count.

**Tech Stack:** PostgreSQL (Supabase migrations + inline `do $$` smoke blocks), Angular 19 standalone components, Vitest (node-env pure-function units), Tailwind.

**Spec:** `docs/superpowers/specs/2026-05-29-unified-recent-change-indicator-design.md`

## Background the engineer must know

- **Never edit an applied migration.** All SQL changes go in ONE new migration created with `supabase migration new`. The two RPCs are changed with `create or replace function`, which requires pasting the *entire current body*. The canonical current bodies live in `supabase/migrations/20260528130000_dashboard_rpcs_emit_trial_acronym.sql`:
  - `get_dashboard_data` — lines 20-226.
  - `get_bullseye_assets` — lines 668-931.
  Copy each verbatim into the new migration, then apply the specific hunks below. Do not change anything else in those bodies.
- **Verifying SQL:** this repo's idiom is an inline `do $$ ... raise exception 'FAIL ...' ... $$;` smoke block at the end of the migration, exercised by `supabase db reset` (which re-runs every migration + seed). A passing reset with the expected `notice` lines is the test.
- **Local Supabase must be running** for `supabase db reset` and `npm run docs:arch`: `supabase start`.
- **Catalysts need no RPC change** — `landscape-state.service.ts:392-393` maps the dashboard trial's `recent_changes_count` into `trial_recent_changes_count`, so updating `get_dashboard_data` covers the catalyst table automatically.
- **`primary_intelligence`** columns used: `space_id`, `entity_type` (`'asset' | 'trial' | ...`), `entity_id`, `state` (`'draft' | 'published'`), `updated_at`. The existing `intelligence_count` field on the bullseye counts ALL intel (drafts included) and is OUT OF SCOPE — leave it untouched.
- **`trial_change_events`** is append-only: `trial_id`, `space_id`, `event_type varchar(40)`, `source varchar(20)`, `payload jsonb not null`, `occurred_at timestamptz not null`, `observed_at timestamptz not null default now()`.

```sql
  --   recent_changes_count rolls up everything beneath the asset:
  --     - trial_change_events for the asset's trials, plus
  --     - published primary_intelligence about the asset OR its trials.
  asset_activity as (
    select
      fa.asset_id,
      (
        (select count(*)
         from public.trial_change_events e
         join public.trials t on t.id = e.trial_id
         where t.asset_id = fa.asset_id
           and t.space_id = p_space_id
           and e.observed_at >= now() - public.recent_change_window())
        +

         from public.primary_intelligence pi
         where pi.space_id = p_space_id
           and pi.state = 'published'
           and pi.updated_at >= now() - public.recent_change_window()
```

```ts
/**
 * Pure presentational logic for the recent-change badge, extracted so it can be
 * unit-tested without Angular DI (see change-badge.logic.spec.ts). The badge is
 * a single neutral slate dot; "recent" means the unified 14-day window computed
 * server-side. See
 * docs/superpowers/specs/2026-05-29-unified-recent-change-indicator-design.md.
 */
export const BADGE_TYPE_LABELS: Record<string, string> = {
  status_changed: 'Status changed',
  date_moved: 'Date moved',
  phase_transitioned: 'Phase transitioned',
  enrollment_target_changed: 'Enrollment target changed',
  arm_added: 'Arm added',
  arm_removed: 'Arm removed',
  intervention_changed: 'Intervention changed',
  outcome_measure_changed: 'Outcome measure changed',
  sponsor_changed: 'Sponsor changed',
  eligibility_criteria_changed: 'Eligibility criteria changed',
  eligibility_changed: 'Eligibility changed',
  trial_withdrawn: 'Trial withdrawn',
  marker_added: 'Marker added',
  projection_finalized: 'Projection finalized',
  marker_reclassified: 'Marker reclassified',
  marker_updated: 'Marker updated',
  marker_removed: 'Marker removed',
  intelligence_published: 'New intelligence',
};

export function badgeTypeLabel(type: string | null): string | null {
  if (!type) return null;
  return BADGE_TYPE_LABELS[type] ?? type.replace(/_/g, ' ');
}

export function badgeTooltip(count: number, type: string | null): string {
  if (count <= 0) return '';
  const label = badgeTypeLabel(type);
  const head = label ? `Recent change: ${label}` : 'Recent change';
  if (count === 1) return head;
  const noun = count === 2 ? 'other change' : 'other changes';
  return `${head} (+${count - 1} ${noun})`;
}
```
