Unified Feed: Events + Activity Merge

One page, three source types, one answer to "what happened."

The problem: Today, "what happened" lives in three places: the Events page (human-authored intelligence), the Catalysts/Markers table, and a hidden Activity page (automated CT.gov change detection). A CI professional checking in on Monday morning has to visit two pages and mentally merge them. This prototype shows a single unified feed.

1. Data mapping: how ChangeEvent becomes a FeedItem

The Events page already renders two source types (Event and Marker). Change events become a third: Detected.

7. Merge summary

What gets simpler

  • One page for "what happened" instead of two
  • Activity page route + component deleted
  • No hidden, undiscoverable page
  • One mental model for the CI analyst
  • Source filter replaces page navigation

What stays the same

  • All three database tables (no schema change)
  • ChangeEventService (still used by widget, trial detail, marker history)
  • Change event row component (still used by widget)
  • What-changed widget data source (still ChangeEventService)
  • Sidebar and header nav (Events was already there)
  • Trial-detail activity card
  • Change badge component

New work

  • RPC: add UNION leg to get_events_page_data
  • FeedItem: add 'detected' source_type
  • Events page: third source filter option
  • Detail panel: detected-event branch with promoted analyst note
  • Widget: entity context per row, entity-count summary, annotation indicator
  • Humanized enum labels (Phase 2, Recruiting, etc.)
  • Annotation indicator on feed table rows
  • Query param handling: source= deep links
  • Redirect: /activity → /events?source=detected
Risk check. The Events page currently loads all data in one RPC call with client-side table pagination. Adding change events increases the row count significantly (change events are high-volume). The RPC may need a page-size limit or server-side pagination to keep load times acceptable. This is the main technical risk of the merge.
