4. Plan -- two tracks

Track A · quick win

Replace native title attributes with pTooltip

Pure cleanup, no design decisions. Eliminates the off-brand OS tooltip and gives the affected copy real keyboard discoverability. Smallest possible PR.

  1. Audit every native title= on interactive elements (grep title=" in templates).
  2. Replace with pTooltip; import TooltipModule in the host components.
  3. Pick position per the convention (nav: right, form: top, content: bottom).
  4. Keep title= only on truly-decorative svg/img where alt text doesn't fit.
  5. Add a one-liner to CLAUDE.md banning native title= on interactive elements.

Track B · structural

Extract <app-hovercard> and refactor landscape tooltips onto it

Real architecture work. Turns three components into one shell + three slot configurations. Pulls the landscape tooltips on-brand and removes a future divergence point.

  1. Create shared/components/hovercard/hovercard.component.ts with slots: accent, eyebrow, title, body, footer.
  2. Move positioning logic (clamp-to-viewport, flip-above) into a useHovercardPosition directive or service so it isn't reimplemented per call site.
  3. Tokenize z-index in one place (e.g. --z-hovercard: 9999); remove inline z-index: 99999.
  4. Refactor marker-tooltip to consume the new shell; verify visual parity in the dashboard grid.
  5. Refactor bullseye-tooltip and positioning-tooltip onto the shell; drop the dark slate-800 surface; add a "Open details" footer link.
  6. Decide animation: keep instant for marker (data scrubbing), keep fade for landscape (lower-frequency hover). Document the choice.
  7. Add a one-pager to docs/runbook/05-frontend-architecture.md: "Hover surfaces -- label tooltip vs hovercard."
