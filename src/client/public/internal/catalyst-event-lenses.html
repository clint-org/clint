Sketch -- Catalyst & event lenses

Same data, three lenses: trial / product / company.

Catalysts and events are trial-scoped in the data model. This sketch tests whether the same underlying rows can earn their place on three different page types by changing the visual idiom and the question each page answers, not by re-cutting the data.

Reference dataset: Eli Lilly / Donanemab (Kisunla) / TRAILBLAZER-ALZ 2. The trial view is a dense chronological table. The product view is a multi-trial development arc with catalysts overlaid on phase bars. The company view is a portfolio matrix where each product is a row and catalysts are dots. Same rows, three frames.

Lens 2 -- Product: answers "where is this asset in its arc?" Every trial becomes a row; phase bars carry the lifecycle; catalysts are dots placed where they occurred in time. The eye picks up the program-level shape (Ph2 PoC, pivotal, approval, prevention extension) in one pass. Events stop being a feed and become a histogram by type, because at the product level the user wants pattern, not transcript.

Verdict on Lens 3 -- recommend Variant B. The matrix (A) is the Evaluate Pharma / BioCentury idiom and reads as portfolio-native, but it doesn't actually add information the timeline can't. It's a simplification: it loses continuous date precision, drops trial granularity, hides phase progression. The timeline (B) with company-scope defaults -- forward 8 quarters, products-as-rows, heavier year gridlines, de-emphasized phase underlines -- recovers most of the matrix's skim-ability while reusing the existing timeline-view component. One chart, three scopes (trial / product / company), three default windows. That collapses the build, keeps the visual grammar consistent across pages, and lets the same export / a11y / legend ride along everywhere.
Open calls

What's still unresolved

1. Settled: one component, three scopes. Lens 3 Variant B comparison confirms the existing timeline-view component carries all three pages with scope params + per-page default time window. The matrix in Variant A reads well but doesn't earn its build cost. Trial page keeps the existing "Markers" table as the primary detail surface; timeline added above is optional and probably skipped.
2. Default time window per scope. Trial: full history. Product: program span (first trial start through last projected catalyst). Company: forward 8 quarters from today. All three should expose a window toggle, but the defaults are the load-bearing decision because most users won't toggle.
3. Scope param shape on timeline-view. Need to read the component to decide whether to add a scope input ('trial' | 'product' | 'company') plus scopeId, or whether to push filtering into LandscapeStateService and have the component stay scope-agnostic. Affects how product / company detail pages embed it.
4. Phase indication at company scope. Variant B uses a thin underline at row foot (amber=approved, teal=P3, cyan=P2). De-emphasized but still readable. Open call: do we render the underline as a real phase bar (with start/end dates) so it's also informative at horizon edges, or keep it as a flat status indicator?
5. Events at company level. Heatmap of event volume by product is the right idiom for "is the company busy?" but offers no drill-through. Click a cell -> filtered activity feed scoped to product+month. Needs the activity feed to accept those filters; check if it already does.
6. Placement on the existing pages. Product detail: timeline goes above the existing intelligence block, since it's the new primary read for the page. Company detail: timeline goes top-of-page, intelligence block below. Trial detail: no change to ordering -- markers table stays where it is.
